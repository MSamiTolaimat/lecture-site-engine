## ترتيب المحتوى (Content Ordering Rules)

### لمادة تحسين الاستعلامات: `type: "diagram-first"` + `prose-second`

هذه المادة تمزج بين:
- **الرسوم البيانية والمخططات** (query trees, transformations)
- **النصوص الشارحة** (المفاهيم والآليات)
- **الخوارزميات والخطوات** (التحسين خطوة بخطوة)

**الترتيب الإلزامي لكل قسم (`### 1.1`):**

1. العنوان + metadata
2. 📍 أين نحن الآن؟
3. ⬅️ الربط مع السابق
4. 💡 الفكرة الأساسية
5. **📊 الرسم البياني / المخطط** (إن وُجد) ← **يأتي أولاً**
   - جدول العُقد (nodes)
   - جدول الروابط (edges)
   - شرح "اقرأ المخطط كالتالي..."
6. 📖 الشرح اللفظي (اشرح المخطط والمفهوم بـ 2-4 فقرات)
7. ⚙️ الخطوات / الخوارزمية (إن لزم)
8. 🎯 الملخص السريع (3-5 نقاط مهمة)
9. 📚 التطبيق (كيف يستخدم هذا في الخطوة التالية؟)
10. ⚠️ أخطاء شائعة
11. 📄 النص الأصلي من المحاضرة (collapsible)

**مثال صغير:**
```markdown
### 1.2. Query Trees (أشجار الاستعلامات)
<!-- @render: {type: "diagram-first", visualization: "tree", coverage: "95%"} -->
<!-- @connectivity: {prerequisite: "section_1.1"} -->

#### 💡 الفكرة الأساسية
**`query tree` هو تمثيل بصري للاستعلام يعكس ترتيب العمليات المراد تنفيذها**

#### 📊 المخطط
```diagram
type: tree
root:
  id: projection
  label: π [P.Pnumber, E.Lname]
  children:
    - id: join1
      label: ⨝ P.Dnum=D.Dnum
      children:
        - id: join2
          label: ⨝ P.Pnumber=W.Pno
        - id: dept
          label: DEPARTMENT (D)
```

#### وصف العُقد:
| العُقدة | النوع | الشرح |
| --- | --- | --- |
| π | projection | تحديد الأعمدة المطلوبة |
| ⨝ | join | ربط الجداول |

#### 📖 الشرح
`query tree` يبدأ من الأسفل (الجداول الأساسية) ويصعد لأعلى (النتيجة النهائية). كل عُقدة تمثل عملية جبرية (`select`, `project`, `join`)، وترتيب العُقد يحدد ترتيب التنفيذ.

#### 🎯 الملخص السريع
- الأوراق (leaves) = الجداول الأساسية
- العُقد الداخلية = العمليات الجبرية
- الجذر (root) = النتيجة النهائية
```

## قواعس الكتل داخل الشرح

### 💻 **الكود / شبه الكود** (SQL أو Relational Algebra)

```markdown
#### 💻 الكود: مثال على استعلام

**ما هذا الكود؟**
> يبحث عن جميع الموظفين من قسم معين مع معلومات المشاريع.

```sql
-- SQL Query
SELECT E.Fname, E.Lname, P.Pname
FROM EMPLOYEE E, WORKS_ON W, PROJECT P
WHERE E.Ssn = W.Essn AND W.Pno = P.Pnumber AND E.Dno = 5;
```

**شرح كل سطر:**
1. `SELECT E.Fname, E.Lname, P.Pname` → اختر اسم الموظف واسم المشروع
2. `FROM EMPLOYEE E, WORKS_ON W, PROJECT P` → من ثلاث جداول
3. `WHERE ...` → مع شروط الربط

**الناتج المتوقع:**
> جدول يحتوي على أسماء الموظفين والمشاريع التي يعملون عليها.
```

### ⚙️ **الخوارزمية / الخطوات**

```markdown
#### ⚙️ الخطوات: تحسين استعلام باستخدام Heuristics

**ما هدف هذه العملية؟**
> تحويل `query tree` غير كفؤ إلى `query tree` محسّن بتطبيق قواعس الدفع.

```algorithm
1 | اقرأ الاستعلام الأصلي | Parser | ينتج `query tree` أولي
2 | اطبّق قاعدة دفع الـ selection | Transformation | يقلل البيانات
3 | اطبّق قاعدة دفع الـ projection | Transformation | يقلل الأعمدة
4 | أعد ترتيب الـ joins | Optimization | يقلل التكلفة
```

**نقاط التنفيذ:**
- يجب دفع الـ selection قبل الـ join دائماً
- يجب دفع الـ projection قبل الـ join إن لم تؤثر على الشرط
```

### 📊 **المخطط / الرسم البياني**

```markdown
#### 📊 المخطط: تحويل Query Tree

**ما هذا المخطط؟**
> يوضح كيفية تحويل شجرة استعلام من الشكل الأول (غير محسّن) إلى الشكل الثاني (محسّن).

**وصف العُقد:**
| # | العُقدة | النوع | الشرح |
| --- | --- | --- | --- |
| 1 | π | Projection | تحديد الأعمدة |
| 2 | σ | Selection | تصفية الصفوف |
| 3 | ⨝ | Join | ربط جدولين |

**وصف الروابط:**
| من | إلى | التسمية | الملاحظات |
| --- | --- | --- | --- |
| σ | ⨝ | input | النتيجة من selection تصير input للـ join |

```diagram
type: tree
root:
  id: proj
  label: π [Fname, Pname]
nodes:
  - id: sel
    label: σ [Dno=5]
  - id: join
    label: ⨝ [Essn=Ssn]
relationships:
  - from: proj
    to: sel
  - from: sel
    to: join
```
```

### 🔍 **تتبع التنفيذ (Trace)**

```markdown
#### 🔍 تتبع التنفيذ: تحويل الاستعلام خطوة بخطوة

**المدخل:** SQL Query `SELECT ... WHERE ... JOIN ...`

**خطوات التحويل:**
| الخطوة | العملية | النتيجة |
| --- | --- | --- |
| 1 | Parse SQL → Query Tree | شجرة أولية (غير محسّنة) |
| 2 | Apply Selection Push | نقل الـ selection للأسفل |
| 3 | Apply Projection Push | نقل الـ projection للأسفل |
| 4 | Reorder Joins | تغيير ترتيب الـ joins |
| 5 | Final Optimized Tree | شجرة محسّنة |

**النتيجة النهائية:** استعلام أكثر كفاءة
```

### 💡 **التشبيه**

```markdown
#### 💡 التشبيه:
> `query tree` أشبه بـ **وصفة طبخ**: كل خطوة تعتمد على الخطوة السابقة، والترتيب مهم جداً. تحسين الاستعلام أشبه بـ **إعادة ترتيب خطوات الطبخ** لتوفير الوقت والجهد.
> 
> **وجه الشبه:** السيئ = تقطيع المكونات مرتين، الجيد = تقطيعها مرة واحدة وحفظها → نفس الفكرة مع البيانات في الاستعلام
```

### ⚖️ **المقايضة (Trade-Off)**

```markdown
#### ⚖️ المقايضة: Heuristics vs Cost-Based Optimization

| المعيار | Heuristics | Cost-Based |
| --- | --- | --- |
| **السرعة** | سريع جداً ✓ | أبطأ (يحسب التكاليف) |
| **الكفاءة** | جيد في الحالات العادية | أفضل للحالات المعقدة ✓ |
| **التعقيد** | بسيط جداً ✓ | معقد |
| **متى تستخدمه؟** | استعلامات بسيطة ومتوسطة | استعلامات معقدة جداً |
```

### 🔄 **قبل / بعد (Before-After)**

```markdown
#### 🔄 قبل / بعد: تحسين الاستعلام

**قبل التحسين (غير كفؤ):**
```
π[E.Fname, P.Pname]
  ⨝[E.Dno=D.Dnum]
    ⨝[P.Pnumber=W.Pno]
      EMPLOYEE × WORKS_ON × PROJECT
```

**بعد التحسين (محسّن):**
```
π[Fname, Pname]
  ⨝[P.Pnumber=W.Pno]
    σ[Dno=5]
      EMPLOYEE
    WORKS_ON
  PROJECT
```

**ماذا تغيّر؟**
> تم نقل الـ selection من الأعلى إلى الأسفل (قبل الـ join)، مما يقلل عدد الصفوف المراد ربطها من الملايين إلى الآلاف.
```

### ⚠️ **الأخطاء الشائعة (Compare)**

```markdown
#### ❌ الفهم الخاطئ
> التحسين بناءً على القواعس دائماً ينتج أفضل خطة تنفيذ

#### ✅ الفهم الصحيح
> التحسين بناءً على القواعس ينتج خطة جيدة في معظم الحالات، لكن قد تكون هناك حالات استثنائية حيث التحسين بناءً على التكلفة أفضل
```

### 📍 **النقاط المهمة (Callouts)**

```markdown
#### مهم للامتحان ⚠️:
> **التمييز بين `query tree` و `query graph`:** الأول يحدد ترتيب العمليات (مهم للتنفيذ)، الثاني لا يحدد الترتيب (مهم للتحليل الأولي).

#### نقطة مهمة ⚠️:
> **دفع الـ selection قبل الـ join أم لا؟** الإجابة تعتمد على شروط الـ selection: إذا كانت تشمل أعمدة من الجدول فقط، ادفعها. إذا كانت تشمل أعمدة من أكثر من جدول، لا تدفعها قبل الـ join.

#### ملاحظة:
> في المحاضرة تم استخدام Q2 كمثال طوال الوقت. تأكد من فهمك للمثال لأنه سيظهر في الامتحان.

#### الدرس المستفاد:
> تحسين الاستعلامات ليس سحراً — هو تطبيق قواعس معقولة (heuristics) تبني على الحدس والتجربة.
```

> **لماذا هذا مهم؟** لأنه يوضح هل فهمت القاعدة الأساسية (دفع selection قبل join يقلل البيانات).
```

### 📐 **المعادلات والرموز**

```markdown
#### 📐 المعادلة: Cascade of σ (تسلسل العمليات)

$$\sigma_{c_1 \land c_2 \land \ldots \land c_n}(R) = \sigma_{c_1}(\sigma_{c_2}(\ldots(\sigma_{c_n}(R))\ldots))$$

**الشرح:**
- **σ:** عملية الـ selection
- **c₁, c₂, ..., cₙ:** شروط مختلفة
- **R:** الجدول الأصلي

**المعنى:** يمكن تقسيم عملية selection واحدة معقدة إلى عمليات متسلسلة بسيطة.
```

## ملاحظات خاصة لمادة تحسين الاستعلامات

1. **الدكتور يستخدم Q2 كمثال أساسي طوال المحاضرات** — تأكد من فهمك لـ:
   - تحويل SQL → Query Tree
   - تطبيق قواعس التحويل خطوة بخطوة
   - الفرق بين الشجرة الأولية والشجرة المحسّنة

2. **الرسوم البيانية مهمة جداً** — كل رسم يجب أن:
   - يبدأ من البيانات الخام (الجداول)
   - ينتهي بالنتيجة النهائية
   - يوضح الفروقات بين الحالات المختلفة

3. **الترتيب الهرمي للأقسام يجب أن يعكس الترتيب الطبيعي**:
   - المقدمة والمفاهيم الأساسية أولاً
   - الخوارزميات والقواعس ثانياً
   - الأمثلة المتقدمة ثالثاً

4. **التغطية (Coverage)** حرجة — إذا كان أقل من 90%، اشرح في `@missing-pieces` لماذا لم تغطِ النقطة بالكامل

5. **استخدم التشبيهات الكثيرة** — هذه مادة تجريدية (abstract) لذا التشبيهات بالحياة اليومية مهمة جداً
