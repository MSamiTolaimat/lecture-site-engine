## الجزء الأول: الشرح التفصيلي (سطر بسطر / فقرة بفقرة)

### 1. الطريقة العامة (The General Method)

#### النص الأصلي يقول:
> "يتم بناء الحل عن طريق مراحل، في كل مرحلة نتخذ قراراً يبدو الأفضل في الوقت الحالي وفق دالة اختيار (Select) ثم نتحقق من الجدوى (Feasible) قبل ضمّه إلى الحل النهائي."

#### الشرح المبسّط:
تخيّل أنك تتسوّق بميزانية محدودة، وفي كل خطوة تختار أرخص وأفضل عنصر متاح **الآن** دون التفكير في الخطوات القادمة، وتستمر في الشراء ما دام المبلغ يسمح. هذا بالضبط ما تفعله `greedy method`: **قرار محلي أفضل في كل خطوة، على أمل أن يؤدي مجموع القرارات المحلية إلى حل عام جيد (وليس بالضرورة الأمثل دائماً)**.

**لماذا؟** لأن حساب كل الاحتمالات الممكنة (`Brute Force`) مكلف زمنياً جداً، فنستبدله باستراتيجية بسيطة وسريعة تعطي حلاً جيداً أو حتى أمثل في كثير من المسائل (خصوصاً عند وجود بنية رياضية مناسبة كالـ `matroid`)، رغم أنها لا تضمن الحل الأمثل في كل الحالات.

#### 💡 التشبيه:
> اختيار أكبر قطعة نقدية ممكنة عند إرجاع الباقي (بدل عدّ كل التركيبات الممكنة).
> **وجه الشبه:** أكبر قطعة نقدية متاحة الآن = العنصر `x` الذي يختاره `Select` في كل خطوة.

#### ⚙️ الخطوات / الخوارزمية: General Greedy Algorithm

> **ما هدف هذه العملية؟** بناء حل مرحلياً بإضافة عنصر واحد في كل خطوة بعد التأكد من جدواه، للوصول لحل مقبول للمسألة (`feasible solution`) يحقّق أو يقارب القيمة المثلى لدالة الهدف (`objective function`).

```algorithm
1 | بدء | solution | تهيئة الحل solution = Empty
2 | تكرار n مرة | Select(a) | اختيار عنصر x من المدخلات a[] يبدو الأفضل حالياً
3 | فحص الجدوى | Feasible(solution, x) | التحقق هل يمكن إضافة x دون كسر قيود المسألة
4 | الدمج | Union(solution, x) | إذا كان x مجدياً، يُضم إلى solution
5 | إرجاع النتيجة | return | إعادة solution بعد انتهاء كل التكرارات
```

#### نقاط التنفيذ:
- `Select` هي التي تحدد "الطمع" الفعلي — تفاصيلها تختلف باختلاف المسألة (مثال: أعلى `p/w` في `Knapsack`).
- `Feasible` لا تُدرج العنصر النهائي في `solution` إلا بعد التأكد أنه لن يخالف القيد (مثال: سعة الحقيبة).
- الخوارزمية **لا ترجع أبداً عن قرار سابق** (لا `backtracking`)، لذلك سرعتها عالية لكنها قد لا تصل للحل الأمثل في كل المسائل.

#### 💻 الكود: Greedy — General Pseudocode

#### ما هذا الكود؟
> هذا القالب المجرد (`Template Method`) الذي تُبنى عليه كل خوارزميات `greedy` اللاحقة في المحاضرة (`Knapsack`, `Optimal Merge`, ...).

```cpp
// SolType greedy(Type a[], int n)
// a[1:n] contains the n inputs
SolType greedy(Type a[], int n) {
    SolType solution = Empty;        // initialize the solution
    for (int i = 1; i <= n; i++) {   // loop over all inputs
        Type x = Select(a);          // pick the locally best candidate
        if (Feasible(solution, x))   // check if adding x keeps solution valid
            solution = Union(solution, x); // merge x into the solution
    }
    return solution;                 // return the final solution
}
```

#### شرح كل سطر:
1. `SolType solution = Empty;` → تهيئة — البداية بحل فارغ قبل أي إضافة.
2. `for (int i = 1; i <= n; i++)` → تكرار — المرور على كل المدخلات المحتملة مرة واحدة فقط.
3. `Type x = Select(a);` → اختيار — يحدد العنصر الأفضل محلياً وفق معيار محدد للمسألة.
4. `if (Feasible(solution, x))` → تحقق — يمنع إضافة عنصر يكسر قيود المسألة.
5. `solution = Union(solution, x);` → دمج — إضافة العنصر المقبول إلى الحل الجاري بناؤه.
6. `return solution;` → إرجاع — الحل النهائي بعد انتهاء كل الدورات.

**الناتج المتوقع (لقطة الشاشة):**
> متجه/مجموعة تمثل `solution` (مثال: مجموعة عناصر مختارة تحقق قيداً معيناً).

#### 🤔 تفعيل الفهم (اسأل نفسك):
> **سؤال:** هل يضمن `greedy` دائماً الحل الأمثل؟
> **لماذا هذا مهم؟** لأن بعض المسائل (كـ `0/1 Knapsack`) لا تُحل بشكل أمثل بـ `greedy`، بينما `Fractional Knapsack` تُحل بشكل أمثل به — الفرق بينهما امتحان أساسي.

### 2. مسألة الحقيبة (Knapsack Problem — النسخة الكسرية)

#### النص الأصلي يقول:
> "لدينا n صنفاً، كل صنف `i` له وزن `wᵢ` وربح `pᵢ`، وسعة الحقيبة `m`. المطلوب إيجاد `xᵢ ∈ [0,1]` (يمكن أخذ جزء من الصنف) بحيث `maximize Σpᵢxᵢ` مع القيد `Σwᵢxᵢ ≤ m`."

#### الشرح المبسّط:
تخيّل حقيبة سفر بسعة محددة بالكيلوغرام، وأمامك عدة أصناف (ذهب، فضة، ألماس...) كل صنف له وزن وربح، ويمكنك أخذ **جزء** من أي صنف (مثلاً نصف كيلو ذهب). المطلوب ملء الحقيبة بأكبر ربح ممكن.

**لماذا؟** لأن القرار الأمثل هنا **لا يعتمد على أخذ الأصناف الأثقل ربحاً فقط، ولا الأخف وزناً فقط**، بل على **نسبة الربح للوزن (`pᵢ/wᵢ`)** — وهذا ما يوضحه المثال العددي التالي.

#### 📐 المعادلة: صياغة مسألة Knapsack

$$
\text{maximize} \sum_{1 \le i \le n} p_i x_i
$$

**الشرح:**
> الهدف هو تعظيم مجموع الأرباح الناتجة عن اختيار كل صنف بنسبة `xᵢ`.

```math
\text{subject to} \sum_{1 \le i \le n} w_i x_i \le m, \quad 0 \le x_i \le 1, \quad 1 \le i \le n
```

**الشرح:**
> `m` = سعة الحقيبة. `wᵢ` = وزن الصنف `i`. `xᵢ` = النسبة المأخوذة من الصنف `i` (بين 0 و1، حيث 1 يعني أخذ الصنف كاملاً).

#### مثال عملي (كما ورد في المحاضرة):
`n=3`، `m=20`، الأوزان `(w₁,w₂,w₃) = (18, 15, 10)`، الأرباح `(p₁,p₂,p₃) = (25, 24, 15)`.

| # | `(x₁,x₂,x₃)` | `Σwᵢxᵢ` | `Σpᵢxᵢ` |
| --- | --- | --- | --- |
| 1 | `(1/2, 1/3, 1/4)` | 16.5 | 24.25 |
| 2 | `(1, 2/15, 0)` | 20 | 28.2 |
| 3 | `(0, 2/3, 1)` | 20 | 31 |
| 4 | `(0, 1, 1/2)` | 20 | **31.5 (الأفضل)** |

#### الشرح المبسّط للجدول:
لاحظ أن الحل الرابع هو الأفضل رغم أنه لا يأخذ الصنف الأول إطلاقاً (`x₁=0`)! هذا لأن نسبة `pᵢ/wᵢ` للصنف الثاني والثالث أعلى من الصنف الأول.

#### 💡 التشبيه:
> لو كنت تملأ سلة تسوّق بميزانية محدودة، تفضّل المنتج الذي يعطيك "قيمة أكبر لكل ريال تدفعه" وليس المنتج الأغلى ثمناً أو الأرخص فقط.
> **وجه الشبه:** نسبة `pᵢ/wᵢ` (الربح لكل وحدة وزن) = "القيمة لكل ريال" في التسوّق.

#### مهم للامتحان ⚠️:
> استراتيجية `greedy` الصحيحة لهذه المسألة هي: **ترتيب الأصناف تنازلياً حسب `vᵢ = pᵢ/wᵢ`** ثم أخذ الأصناف بالكامل بهذا الترتيب حتى تمتلئ الحقيبة، وأخذ **جزء (كسر)** من الصنف الذي يفيض عن السعة.

#### ⚙️ الخطوات / الخوارزمية: Greedy Knapsack

> **ما هدف هذه العملية؟** إيجاد الحل الأمثل لـ `Fractional Knapsack` بأخذ الأصناف مرتبة تنازلياً حسب `pᵢ/wᵢ`.

```algorithm
1 | ترتيب مسبق | sort | ترتيب الأصناف تنازلياً حسب p[i]/w[i]
2 | تهيئة | init | تصفير كل x[i] ووضع U = m (السعة المتبقية)
3 | حلقة على الأصناف | loop | لكل صنف i بالترتيب، إذا كان w[i] > U توقف
4 | أخذ الصنف كاملاً | assign | ضع x[i] = 1.0 واطرح w[i] من U
5 | أخذ الكسر الأخير | fraction | عند توقف الحلقة، ضع x[i] = U / w[i]
```

#### نقاط التنفيذ:
- الترتيب يجب أن يتم **قبل** تشغيل الحلقة الأساسية (تكلفته هي المسيطرة على التعقيد الكلي).
- الكسر يُؤخذ من **صنف واحد فقط** وهو أول صنف لا يتّسع بالكامل.

#### 💻 الكود: greedyKnapsack()

#### ما هذا الكود؟
> تنفيذ فعلي لخوارزمية `Fractional Knapsack` بافتراض أن المصفوفات مرتبة مسبقاً حسب `p[i]/w[i]`.

```cpp
// p[1:n] and w[1:n] contain profits and weights, sorted so that
// p[i]/w[i] >= p[i+1]/w[i+1]. m is knapsack size, x[1:n] is the solution.
void greedyKnapsack(float m, int n) {
    for (int i = 1; i <= n; i++) x[i] = 0.0; // initialize all x to zero
    float U = m;                              // U tracks remaining capacity
    int i;
    for (i = 1; i <= n; i++) {
        if (w[i] > U) break;                  // item doesn't fit fully, stop
        x[i] = 1.0;                            // take whole item i
        U -= w[i];                             // reduce remaining capacity
    }
    if (i <= n) x[i] = U / w[i];               // take fractional part of item i
}
```

#### شرح كل سطر:
1. `for (int i = 1; i <= n; i++) x[i] = 0.0;` → تهيئة — كل الأصناف تبدأ بدون أخذ شيء منها.
2. `float U = m;` → متغير حالة — السعة المتبقية في الحقيبة، تبدأ بالسعة الكلية.
3. `for (i = 1; i <= n; i++) {` → حلقة رئيسية — تمر على الأصناف بالترتيب حسب `p/w`.
4. `if (w[i] > U) break;` → شرط توقف — إذا لم يعد الصنف يتسع، أوقف الأخذ الكامل.
5. `x[i] = 1.0;` → أخذ كامل — الصنف `i` بالكامل يدخل الحقيبة.
6. `U -= w[i];` → تحديث — إنقاص السعة المتبقية بمقدار وزن الصنف المأخوذ.
7. `if (i <= n) x[i] = U / w[i];` → الكسر الأخير — يملأ الفراغ المتبقي بجزء من الصنف التالي.

**الناتج المتوقع (لقطة الشاشة):**
> مصفوفة `x[]` تحتوي 1.0 للأصناف المأخوذة بالكامل، كسراً عشرياً لصنف واحد، و0.0 لبقية الأصناف.

#### نقطة مهمة ⚠️:
> إذا كانت المصفوفات **غير مرتبة مسبقاً**، فإن الترتيب باستخدام خوارزمية فرز فعالة يكلّف `O(n log₂n)`، وهو ما يحدد التعقيد الكلي للخوارزمية.

### 3. تجزئة رؤوس الشجرة (Tree Vertex Splitting — TVSP)

#### النص الأصلي يقول:
> "يُعطى بيان موجّه ذو وزن `T=(V,E,w)`، رأس مصدر `source` ورأس هدف `sink`. تأخير المسار `delay` هو أكبر تأخير `d(p)` لأي مسار `p` في الشجرة. المطلوب: تحديد أقل عدد من الرؤوس (`vertex set X`) بحيث بعد "قصّها" لا يتجاوز `d(T/X)` قيمة `δ` محددة."

#### الشرح المبسّط:
تخيّل شبكة اتصالات على شكل شجرة، كل وصلة (`edge`) لها تأخير زمني، والمسار من المصدر (`source`) إلى أي نقطة (`sink`) يجمع تأخيرات وصلاته. إذا أصبح مسار ما بطيئاً جداً (تجاوز حداً مسموحاً `δ`)، نحتاج "لقطع" الشجرة عند بعض الرؤوس (كأننا نضع "مُكرِّراً" يعيد ضبط التأخير إلى صفر) بحيث لا يتجاوز أي مسار الحد `δ` بعد القطع، مع استخدام **أقل عدد ممكن** من نقاط القطع.

**لماذا؟** لأن وضع "مُكرِّر" في كل نقطة مكلف (تكلفة مادية أو حسابية)، لذا نريد التقليل لأدنى عدد ممكن من نقاط القطع (`min cardinality split set`) يحقق الشرط.

#### 💡 التشبيه:
> كأنك تضع محطات تقوية إشارة (`repeaters`) في أنابيب مياه طويلة تفقد ضغطها تدريجياً — تضع أقل عدد من المحطات في أنسب الأماكن لإبقاء الضغط ضمن الحد المسموح.
> **وجه الشبه:** التأخير المتراكم `d(p)` = فقدان الضغط، رأس القطع `X` = محطة إعادة الضغط.

#### 📐 المعادلة: تأخير المسار

$$
d(u) = \max_{w \in C(u)} \{ d(w) + w(w,u) \}
$$

**الشرح:**
> `d(u)` = التأخير الأقصى الواصل عند الرأس `u`. `C(u)` = مجموعة أبناء `u` في الشجرة. `w(w,u)` = وزن الوصلة بين `w` و`u`. إذا كان `u` ورقة (`leaf`) فإن `d(u)=0`.

#### مثال عملي (كما ورد في المحاضرة، `δ=5`):
التأخيرات على الحواف: `7, 8, 5, 9, 0`. عند حساب `d(4)=0` (ورقة)، ثم فحص `d(2)+w(2,4)=4+2=6 > δ` → يُقصّ الرأس `4`. وبالمثل يُقصّ الرأسان `2` و`6`، فتصبح مجموعة القطع `{2, 4, 6}`.

#### 🔍 تتبع التنفيذ: حساب التأخير وقصّ الرؤوس (δ=5)

**المدخل:** شجرة بأوزان الحواف `7, 8, 5, 9, 0` وقيمة `δ = 5`

| الخطوة | العملية | الحالة |
| --- | --- | --- |
| 1 | حساب `d` للورقة 4 | `d(4) = 0` |
| 2 | فحص `d(4)+w(2,4) > δ` | `4+2=6 > 5` → قصّ الرأس 4، `d(4)=0` بعد القص |
| 3 | فحص `d(2)+w(1,2) > δ` | يتحقق → قصّ الرأس 2 |
| 4 | فحص `d(6)+w(3,6) > δ` | `d(6)=3`، `6 > 5` → قصّ الرأس 6 |

**النتيجة:** مجموعة القطع النهائية = `{2, 4, 6}`

#### ⚙️ الخطوات / الخوارزمية: TVS (Recursive, Tree Pointer)

> **ما هدف هذه العملية؟** تحديد وإخراج كل الرؤوس التي يجب "قصّها" (`split`) بحيث لا يتجاوز أي مسار من الجذر التأخير `δ`، باستخدام أقل عدد من نقاط القطع، عبر مرور `post-order` على الشجرة.

```algorithm
1 | حالة الأساس | if(T!=NULL) | إذا كانت الشجرة غير فارغة، تابع الحساب
2 | تهيئة التأخير | d[T]=0 | ابدأ تأخير الرأس T بصفر
3 | مرور على الأبناء | for each child v | لكل ابن v من T، استدعِ TVS(v, delta) بشكل تكراري
4 | تحديث التأخير | d[T]=max(...) | حدّث d[T] = أكبر قيمة بين d الحالي و(d[v]+w(T,v))
5 | فحص تجاوز الحد | if d[T]+w(parent,T) > delta | تحقق هل تأخير T (بعد ربطه بالأب) يتجاوز δ
6 | القص والإخراج | cout << T; d[T]=0 | إذا تجاوز: أخرج T كنقطة قص وصفّر تأخيره
```

#### نقاط التنفيذ:
- المرور `post-order` **إلزامي**: يجب حساب تأخير الأبناء أولاً قبل الأب.
- الجذر (`root`) لا يُفحص للقص لأنه ليس له أب.
- بعد القص، يُعاد تصفير `d[T]` لأن أي "قطع" يُعيد ضبط التأخير المتراكم إلى صفر (كأنه بداية جديدة).

#### 💻 الكود: TVS (Tree Pointer Version)

#### ما هذا الكود؟
> تنفيذ `TVS` باستخدام مؤشرات الشجرة (`tree-pointer`) بدلاً من التمثيل التسلسلي.

```cpp
// Determine and output the nodes to be split; w() is edge weight function
void TVS(tree_pointer T, Type delta) {
    if (T != NULL) {                              // proceed only if node exists
        d[T] = 0;                                  // start delay at zero
        for (/* each child v of T */) {             // visit every child
            TVS(v, delta);                          // recurse into child first
            d[T] = max(d[T], d[v] + w(T, v));       // update T's delay
        }
        if ((T is not the root) &&
            (d[T] + w(parent(T), T) > delta)) {     // check threshold
            cout << T << endl;                      // output split vertex
            d[T] = 0;                                // reset delay after split
        }
    }
}
```

#### شرح كل سطر:
1. `if (T != NULL)` → حارس — يمنع المعالجة على شجرة/فرع فارغ.
2. `d[T] = 0;` → تهيئة — تأخير الرأس يبدأ من الصفر قبل جمع تأخيرات الأبناء.
3. `for (each child v of T)` → تكرار — يزور كل الأبناء واحداً تلو الآخر.
4. `TVS(v, delta);` → استدعاء تكراري (`post-order`) — يحسب تأخير الابن أولاً.
5. `d[T] = max(d[T], d[v] + w(T, v));` → تحديث — يأخذ أسوأ (أكبر) تأخير قادم من الأبناء.
6. `if (...) > delta` → شرط — يفحص هل التأخير المتراكم مع وزن الوصلة للأب يتجاوز الحد.
7. `cout << T; d[T] = 0;` → إجراء — إخراج الرأس كنقطة قصّ وإعادة ضبط تأخيره.

**الناتج المتوقع (لقطة الشاشة):**
> قائمة بأرقام/رموز الرؤوس التي يجب قصّها (في المثال: `2, 4, 6`).

#### 💻 الكود: TVS (Sequential Array Representation)

#### ما هذا الكود؟
> نفس الخوارزمية لكن بتمثيل تسلسلي للشجرة (`tree[]` كمصفوفة، الجذر عند `tree[1]`، أبناء العقدة `i` هم `2i` و`2i+1`) — أسلوب شائع لتمثيل الأشجار الثنائية الكاملة دون مؤشرات.

```cpp
// tree[], weight[], d[] use array-based (heap-style) tree representation
// Root at tree[1]; N = index of last valid tree node
void TVS(int i, Type delta) {
    if (tree[i]) {                                  // node i exists
        if (2*i > N) {                               // i has no children -> leaf
            d[i] = 0;
        } else {
            TVS(2*i, delta);                          // recurse left child
            d[i] = max(d[i], d[2*i] + weight[2*i]);   // update from left child
            if (2*i + 1 <= N) {                       // right child exists
                TVS(2*i + 1, delta);                  // recurse right child
                d[i] = max(d[i], d[2*i+1] + weight[2*i+1]); // update from right
            }
        }
        if ((tree[i] != 1) && (d[i] + weight[i] > delta)) { // not root & exceeds
            cout << tree[i] << endl;                  // output split vertex
            d[i] = 0;                                  // reset delay
        }
    }
}
```

#### شرح كل سطر:
1. `if (tree[i])` → حارس — يتأكد أن هذه الخانة تمثل عقدة فعلية في الشجرة.
2. `if (2*i > N)` → فحص ورقة — إذا لم يوجد ابن أيسر، فالعقدة ورقة، تأخيرها صفر.
3. `TVS(2*i, delta);` → استدعاء الابن الأيسر أولاً (`post-order`).
4. `d[i] = max(d[i], d[2*i]+weight[2*i]);` → تحديث تأخير الأب من الابن الأيسر.
5. `if (2*i+1 <= N)` → فحص وجود ابن أيمن قبل معالجته.
6. `TVS(2*i+1, delta); d[i]=max(...)` → نفس المعالجة للابن الأيمن.
7. `if ((tree[i]!=1) && (d[i]+weight[i] > delta))` → شرط: ليس الجذر (`tree[i]!=1`) وتجاوز الحد.
8. `cout << tree[i] << endl; d[i] = 0;` → إخراج رقم العقدة وإعادة تصفير تأخيرها.

**الناتج المتوقع (لقطة الشاشة):**
> نفس مخرجات النسخة بالمؤشرات، لكن بالاعتماد على `tree[] = {1,2,3,0,4,5,6,0,0,7,8,0,0,9,10}` و`weight[]` المقابلة كمثال في المحاضرة.

#### نقطة مهمة ⚠️:
> تعقيد `TVS` هو **`Θ(n)`** لأن كل رأس يُزار مرة واحدة فقط في المرور `post-order` (سواء بالتمثيل بالمؤشرات أو بالمصفوفة).

### 4. أقصر المسارات من مصدر واحد إلى كل الوجهات (Dijkstra's Algorithm)

#### النص الأصلي يقول:
> "يُعطى بيان موجّه وموزون `G=<V,E>` بحيث `w(e) > 0` لكل ضلع، ورأس بداية `v`. الهدف: توليد أطوال المسارات الأقصر من `v` إلى كل بقية الرؤوس بشكل تصاعدي."

#### الشرح المبسّط:
تخيّل أنك تبحث عن أقصر طريق بالسيارة من مدينتك إلى كل المدن الأخرى، وكل طريق بينهما له مسافة (وزن موجب). خوارزمية `Dijkstra` تبني الحل تدريجياً: تحدد أولاً أقرب مدينة، ثم الأقرب بعدها، وهكذا — وفي كل خطوة "تحدّث" تقديراتها لبقية المدن إذا وجدت طريقاً أقصر يمر بالمدينة التي أضافتها للتو.

**لماذا؟** لأن المرور بمدينة معروف أنها الأقرب حالياً **لن يُقصّر أي مسار آخر أبعد منها** (بما أن كل الأوزان موجبة)، لذلك بمجرد تثبيت أقصر مسار لرأس ما، لا حاجة لإعادة النظر فيه — وهذا هو جوهر أسلوب `greedy` هنا.

#### مثال عملي (البيان الموجّه الموزون في الشكل ٣-٤):
إذا كان رأس البداية `1`، فأقصر مسار من `1` إلى `2` هو `1, 3, 4, 2` بطول `10+15+20=45` رغم وجود 3 أضلاع في هذا المسار مقابل ضلع مباشر واحد بوزن `50`.

| Path | Length |
| --- | --- |
| `1, 4` | 10 |
| `1, 4, 5` | 25 |
| `1, 4, 5, 2` | 45 |
| `1, 3` | 45 |

#### 💡 التشبيه:
> كأنك توسّع دائرة "المدن المؤكد أنها الأقرب" تدريجياً من مركزها (نقطة البداية)، وفي كل توسعة تضيف أقرب مدينة لم تُضف بعد.
> **وجه الشبه:** المجموعة `S` (الرؤوس التي وُجد أقصر مسار لها) = دائرة اليقين المتوسعة تدريجياً.

#### 📐 المعادلة: تحديث المسافة (Relaxation)

$$
distance[w] = distance[u] + length(\langle u, w \rangle)
$$

**الشرح:**
> عند اختيار `u` كأقرب رأس تالٍ، نحدّث تقدير `distance[w]` لكل جار `w` لم يُثبّت بعد، إذا كان المسار عبر `u` أقصر من التقدير الحالي.

#### ⚙️ الخطوات / الخوارزمية: Dijkstra — Shortest Path

> **ما هدف هذه العملية؟** إيجاد أقصر مسار من رأس بداية `v` إلى كل بقية رؤوس البيان `G` الموزون بأوزان موجبة.

```algorithm
1 | تهيئة | init | ضع found[i]=FALSE لكل i، و distance[i]=cost[v][i]
2 | تثبيت البداية | found[v]=TRUE | ضع distance[v]=0 لأن المسافة من v لنفسه صفر
3 | حلقة رئيسية n-1 مرة | choose() | اختر u صاحب أصغر distance بين الرؤوس غير المثبتة
4 | تثبيت u | found[u]=TRUE | أعلن أن أقصر مسار لـ u أصبح نهائياً
5 | تحديث الجيران | relax | لكل w غير مثبت، إذا distance[u]+cost[u][w] < distance[w] حدّثها
```

#### نقاط التنفيذ:
- الخوارزمية تفترض **أوزاناً موجبة فقط** (`w(e) > 0`)؛ عند وجود أوزان سالبة تفشل الخوارزمية.
- بعد تثبيت رأس (`found[i]=TRUE`)، **لا يُعاد النظر في مسافته أبداً**.

#### 💻 الكود: shortest_path()

#### ما هذا الكود؟
> التطبيق الفعلي لخوارزمية `Dijkstra` باستخدام مصفوفة تكلفة `cost[][]` (مصفوفة تجاور) ومصفوفتي `distance[]` و`found[]`.

```cpp
// distance[i] = shortest path from v to i so far
// found[i] = whether the shortest path to i is finalized
// cost is the adjacency matrix
void shortest_path(int v, int cost[][MAX_VERTICES], int distance[], int n, short int found[]) {
    int i, u, w;
    for (i = 1; i <= n; i++) {           // initialize all vertices
        found[i] = FALSE;                 // none finalized yet
        distance[i] = cost[v][i];         // direct edge cost (or infinity)
    }
    found[v] = TRUE;                      // starting vertex is trivially finalized
    distance[v] = 0;                      // distance to itself is zero
    for (i = 1; i <= n - 1; i++) {        // repeat for remaining n-1 vertices
        u = choose(distance, n, found);   // pick closest unfinalized vertex
        found[u] = TRUE;                  // finalize u
        for (w = 1; w <= n; w++)          // try relaxing all neighbors
            if (found[w] == FALSE)
                if (distance[u] + cost[u][w] < distance[w])
                    distance[w] = distance[u] + cost[u][w]; // update shorter path
    }
}
```

#### شرح كل سطر:
1. `for (i = 1; i <= n; i++) { found[i]=FALSE; distance[i]=cost[v][i]; }` → تهيئة — كل رأس يبدأ بمسافة الضلع المباشر من `v` (أو ∞ إن لم يوجد ضلع).
2. `found[v] = TRUE; distance[v] = 0;` → حالة خاصة — نقطة البداية مثبتة فوراً بمسافة صفر.
3. `for (i = 1; i <= n - 1; i++)` → حلقة رئيسية — تُنفَّذ `n-1` مرة لتثبيت بقية الرؤوس.
4. `u = choose(distance, n, found);` → اختيار — يجد أقرب رأس غير مثبت بعد.
5. `found[u] = TRUE;` → تثبيت — يُعلن أن مسافة `u` نهائية ولن تتغير.
6. `for (w = 1; w <= n; w++) if (found[w]==FALSE)` → مرور على الجيران غير المثبتين فقط.
7. `if (distance[u]+cost[u][w] < distance[w]) distance[w]=...` → تحديث (`relaxation`) إذا كان المسار عبر `u` أقصر.

**الناتج المتوقع (لقطة الشاشة):**
> مصفوفة `distance[]` نهائية تحتوي أقصر مسافة من `v` إلى كل رأس آخر.

#### 💻 الكود: choose()

#### ما هذا الكود؟
> دالة مساعدة تبحث عن الرأس صاحب أصغر `distance` من بين الرؤوس التي لم تُثبَّت بعد.

```cpp
// find the vertex with smallest distance among unfinalized vertices
int choose(int distance[], int n, short int found[]) {
    int min = INFTY;      // start with the largest possible value
    int minpos = 0;        // will hold the index of the chosen vertex
    for (int i = 1; i <= n; i++)
        if (distance[i] < min && found[i] == FALSE) { // candidate check
            min = distance[i];   // update current minimum
            minpos = i;          // remember its index
        }
    return minpos;          // return chosen vertex
}
```

#### شرح كل سطر:
1. `int min = INFTY;` → تهيئة — قيمة كبيرة جداً لضمان استبدالها بأول قيمة حقيقية.
2. `int minpos = 0;` → تهيئة — لا يوجد رأس مختار بعد.
3. `for (int i = 1; i <= n; i++)` → مرور على كل الرؤوس.
4. `if (distance[i] < min && found[i]==FALSE)` → شرط مزدوج: أصغر مسافة **و**غير مثبت بعد.
5. `min = distance[i]; minpos = i;` → تحديث أفضل مرشح حتى الآن.
6. `return minpos;` → إرجاع فهرس الرأس صاحب أصغر مسافة غير مثبتة.

**الناتج المتوقع (لقطة الشاشة):**
> رقم الرأس (فهرس) صاحب أصغر `distance` بين الرؤوس غير المثبتة، أو `0` إن لم يوجد.

#### نقطة مهمة ⚠️:
> تعقيد `shortest_path()` هو **`O(n²)`**: الحلقة الخارجية `for` تُنفَّذ `n-1` مرة، وفي كل مرة تستدعي `choose()` بتكلفة `O(n)` وتُحدّث الجيران بتكلفة `O(n)` — الإجمالي `O(n) × O(n) = O(n²)` بغض النظر عن ترتيب الرؤوس أو عدد الأضلاع.

### 5. أنماط الدمج المثلى (Optimal Merge Patterns)

#### النص الأصلي يقول:
> "لدينا ملفان مرتّبان بطول `n` و`m`، يمكن دمجهما في `O(n+m)`. عند وجود أكثر من ملفين، ترتيب عمليات الدمج يؤثر على العدد الكلي لعمليات المقارنة/التحرك."

#### الشرح المبسّط:
تخيّل عدة قوائم أسماء مرتّبة أبجدياً بأطوال مختلفة، وتريد دمجها كلها في قائمة واحدة عبر دمج قائمتين في كل مرة. **الترتيب الذي تدمج به القوائم يُغيّر إجمالي عدد الخطوات**! الاستراتيجية الذكية (`greedy`): **ادمج دائماً أصغر قائمتين متاحتين أولاً**.

**لماذا؟** لأن القوائم الصغيرة إذا أُخّر دمجها ستُعاد "زيارتها" أكثر من مرة كجزء من دمجات لاحقة، فتتضاعف تكلفة عناصرها الفردية بشكل غير ضروري.

#### مثال عملي (كما ورد في المحاضرة):
السجلات `x₁,x₂,x₃` بأطوال `10, 20, 30`. دمج `x₁` مع `x₂` (يتطلب 30 تحرك) ثم دمج الناتج مع `x₃` (يتطلب 60 تحرك) = **90 تحركاً إجمالاً**. بينما دمج `x₂,x₃` أولاً (يتطلب 50 تحرك، وهو أسوأ من دمج `x₁,x₂` أولاً) ثم دمج الناتج مع `x₁` (يتطلب 60 تحرك) = **110 تحركات** — أبطأ!

#### مثال الشرهة لخمسة ملفات (5,30,20,10,30) → أنماط التوالي:
1. دمج `x₃,x₄` (`5+10=15`) → `z₁`
2. دمج `z₁,x₁` (`15+20=35`) → `z₂`
3. دمج `x₂,x₅` (`30+30=60`) → `z₃`
4. دمج `z₂,z₃` (`35+60=95`) → `z₄`

العدد الكلي لتحركات السجلات = `205`.

#### 💡 التشبيه:
> عند لملمة أكوام أوراق مختلفة الحجم لتُصبح كومة واحدة، تبدأ دائماً بدمج أصغر كومتين معاً — بدل جرّ الكومة الكبيرة في كل خطوة.
> **وجه الشبه:** أصغر كومتين = أصغر قائمتين متاحتين للدمج حالياً.

#### 📐 المعادلة: طول المسار الخارجي الموزون (Weighted External Path Length)

$$
\sum_{i=1}^{n} d_i q_i
$$

**الشرح:**
> `dᵢ` = المسافة (عمق) العقدة الطرفية `i` عن الجذر. `qᵢ` = حجم الملف `xᵢ` عند تلك العقدة. المجموع الكلي يساوي إجمالي عدد تحركات السجلات في عملية الدمج بالكامل — وهذا ما تُحاول خوارزمية `greedy` تصغيره.

#### ⚙️ الخطوات / الخوارزمية: Tree() — بناء شجرة الدمج المثلى

> **ما هدف هذه العملية؟** بناء شجرة الدمج ذات أقل طول مسار خارجي موزون، بدمج أصغر شجرتين تكراراً حتى تبقى شجرة واحدة.

```algorithm
1 | تهيئة | list | كل ملف يبدأ كشجرة عقدة واحدة (n شجرة) في list
2 | تكرار n-1 مرة | loop | استخرج أصغر شجرتين من list
3 | استخراج الأصغر | Least(list) | أخرج الشجرة صاحبة أصغر وزن مرتين متتاليتين
4 | بناء عقدة جديدة | pt | أنشئ عقدة أب جديدة أبناؤها الشجرتان المستخرجتان
5 | حساب الوزن | weight | وزن العقدة الجديدة = مجموع وزني الابنين
6 | إعادة الإدراج | Insert(list, pt) | أدرج الشجرة الجديدة في list واستمر
```

#### نقاط التنفيذ:
- يجب استخراج أصغر شجرتين **في كل تكرار**، وليس شجرة واحدة فقط.
- إذا استُخدمت `heap` (كومة صغرى) لتمثيل `list`، تصبح `Least` و`Insert` بتكلفة `O(log n)` بدل `O(n)`.

#### 💻 الكود: Tree()

#### ما هذا الكود؟
> بناء شجرة دمج مثلى (`Optimal Merge Tree`) عبر دمج أصغر شجرتين تكراراً — تطبيق مباشر لاستراتيجية `greedy` على مسألة الدمج.

```cpp
struct tree_node {
    struct tree_node *l_child, *r_child; // pointers to left/right subtree
    int weight;                          // total size represented by this node
};
typedef struct tree_node Type;

Type *Tree(int n) {
    // list is a global list of n single-node binary trees
    for (int i = 1; i < n; i++) {                 // repeat n-1 times
        Type *pt = new Type;                       // get a new tree node
        pt->l_child = Least(list);                  // take smallest-weight tree
        pt->r_child = Least(list);                  // take next smallest tree
        pt->weight = (pt->l_child)->weight + (pt->r_child)->weight; // sum weights
        Insert(list, *pt);                           // put merged tree back
    }
    return (Least(list)); // the single remaining tree is the merge tree
}
```

#### شرح كل سطر:
1. `struct tree_node { ... };` → تعريف — كل عقدة تحتوي مؤشرين لأبنائها ووزنها الكلي.
2. `for (int i = 1; i < n; i++)` → حلقة — تُنفَّذ `n-1` مرة لدمج `n` شجرة في شجرة واحدة.
3. `Type *pt = new Type;` → إنشاء — عقدة أب جديدة ستحمل شجرتين مدموجتين.
4. `pt->l_child = Least(list);` → استخراج — أصغر شجرة متاحة حالياً تصبح الابن الأيسر.
5. `pt->r_child = Least(list);` → استخراج — ثاني أصغر شجرة تصبح الابن الأيمن.
6. `pt->weight = ... + ...;` → حساب — وزن الأب هو مجموع وزني ابنيه (لاستخدامه في الدمجات اللاحقة).
7. `Insert(list, *pt);` → إعادة إدراج — الشجرة المدموجة الجديدة تدخل `list` لتشارك في الدمجات القادمة.
8. `return (Least(list));` → إرجاع — بعد `n-1` تكرار تبقى شجرة واحدة، وهي الشجرة النهائية.

**الناتج المتوقع (لقطة الشاشة):**
> شجرة ثنائية واحدة تمثل ترتيب الدمج الأمثل (كما في مثال الملفات الستة `2,3,5,7,9,13`).

#### 🔍 تتبع التنفيذ: Tree() على الملفات (2,3,5,7,9,13)

**المدخل:** `list = {2, 3, 5, 7, 9, 13}` (كل رقم شجرة عقدة واحدة)

| الخطوة | العملية | الحالة (list بعد الخطوة) |
| --- | --- | --- |
| 1 | دمج `2,3` → شجرة وزن `5` | `{5, 5, 7, 9, 13}` |
| 2 | دمج `5,5` → شجرة وزن `10` | `{10, 7, 9, 13}` |
| 3 | دمج `7,9` → شجرة وزن `16` | `{10, 16, 13}` |
| 4 | دمج `10,13` → شجرة وزن `23` | `{23, 16}` |
| 5 | دمج `23,16` → شجرة وزن `39` | `{39}` (شجرة نهائية) |

**النتيجة:** شجرة دمج واحدة بوزن جذر `39` (كما في الشكل ٣-٦ بالمحاضرة).

#### نقطة مهمة ⚠️:
> إذا كانت `list` **مصفوفة عادية مرتّبة**: `Least` بتكلفة `O(1)` و`Insert` بتكلفة `O(n)` → التعقيد الكلي `O(n²)`.
> إذا كانت `list` **كومة صغرى (`min-heap`)**: كلٌ من `Least` و`Insert` بتكلفة `O(log n)` → التعقيد الكلي `O(n log₂n)` — وهو الأفضل.

#### ⚖️ المقايضة: تمثيل list — مصفوفة مرتبة مقابل Heap

| | مصفوفة مرتّبة | كومة صغرى (Heap) |
| --- | --- | --- |
| المزايا | استخراج الأصغر `O(1)` فوري | إدراج واستخراج كلاهما `O(log n)` |
| العيوب | الإدراج يتطلب إزاحة عناصر `O(n)` | تعقيد تنفيذ أعلى قليلاً |
| متى تختاره | عدد الملفات صغير جداً | عدد الملفات كبير (الأداء العام أفضل) |

### 6. مدخل إلى أشجار الامتداد ذات الكلفة الأقل (Minimum-Cost Spanning Trees)

#### النص الأصلي يقول:
> "بفرض `G=<V,E>` بيان غير موجّه وموزون ومترابط ومكوّن من `n` رأس، شجرة الامتداد `t=<V,É>` هي بيان جزئي من `G` يحقق كونها شجرة (لا حلقات) تغطي كل الرؤوس. كلفة شجرة الامتداد = مجموع أوزان أضلاعها."

#### الشرح المبسّط:
تخيّل شبكة طرق تربط عدة مدن (البيان `G`)، وتريد اختيار **أقل مجموعة طرق ممكنة** تُبقي كل المدن متصلة ببعضها (شجرة امتداد)، وبأقل كلفة إجمالية ممكنة (شجرة الامتداد ذات الكلفة الأقل).

**لماذا؟** لأن مد كل الطرق (كل الأضلاع) مكلف جداً، ونحتاج فقط لأقل عدد من الأضلاع (`n-1` بالضبط) يُبقي الشبكة متصلة دون حلقات زائدة، وباختيار الأضلاع الأرخص قدر الإمكان.

#### القيود الإلزامية لشجرة الامتداد:
- الأضلاع المستخدمة يجب أن تكون فقط من البيان الأصلي `G`.
- عدد الأضلاع المستخدمة = `n-1` بالضبط (حيث `n` عدد الرؤوس).
- لا يُستخدم أي ضلع يُشكّل حلقة (`cycle`) مع الأضلاع المختارة سابقاً.

#### 💡 التشبيه:
> كأنك تصمم شبكة أنابيب مياه تربط كل الأحياء بأقل طول أنابيب ممكن، دون أي حلقة زائدة تكرر الاتصال بين نفس الأحياء.
> **وجه الشبه:** الأحياء = الرؤوس، الأنابيب = الأضلاع، تجنّب الحلقات = عدم إهدار مواد إضافية.

#### مهم للامتحان ⚠️:
> ذُكرت خوارزميتا `Kruskal` و`Prim` كخوارزميتين مصمّمتين بتقنية `greedy` لحل هذه المسألة، وتستخدمان معيار "الأقل كلفة" (`least cost`) في كل خطوة اختيار.

#### ⚙️ الخطوات / الخوارزمية: مبدأ Kruskal (تمهيدي كما ورد في المحاضرة)

> **ما هدف هذه العملية؟** بناء شجرة امتداد ذات كلفة أقل عبر اختيار الأضلاع الأرخص تدريجياً، شرط عدم تكوين حلقة.

```algorithm
1 | ترتيب الأضلاع | sort | رتّب كل أضلاع البيان G تصاعدياً حسب الوزن
2 | اختيار الأرخص | select | اختر الضلع الأرخص غير المُختار بعد
3 | فحص الحلقة | check cycle | تحقق: هل إضافة هذا الضلع تُكوّن حلقة مع الأضلاع المختارة؟
4 | القبول أو الرفض | accept/reject | إن لم تتكوّن حلقة اقبل الضلع، وإلا ارفضه وتابع للضلع التالي
5 | التوقف | stop | توقف عند اختيار n-1 ضلعاً بالضبط
```

#### نقطة مهمة ⚠️:
> هذا الجزء ورد كمدخل نظري في المحاضرة (تعريف المسألة والقيود)، والتفاصيل الكاملة لخوارزميتي `Kruskal` و`Prim` (بما فيها الكود والتحليل) لم تُشرح بالتفصيل الكامل في هذا المقطع من المحاضرة (**غير مشروحة بالتفصيل في المحاضرة**) — سيتم تناولها لاحقاً.

## الجزء الثاني: ملخص منظم

### أهم التعاريف والمفاهيم
| المصطلح | التعريف | مثال/ملاحظة |
| --- | --- | --- |
| `Greedy Method` | بناء الحل بقرارات محلية أفضل في كل خطوة دون رجوع | اختيار `p/w` الأعلى في Knapsack |
| `Feasible Solution` | حل جزئي يحقق كل قيود المسألة حتى الآن | `Σwᵢxᵢ ≤ m` |
| `Objective Function` | الدالة التي نُريد تعظيمها أو تصغيرها | `maximize Σpᵢxᵢ` |
| `Fractional Knapsack` | نسخة من مسألة الحقيبة تسمح بأخذ جزء من الصنف | تُحل بشكل أمثل بـ `greedy` |
| `Tree Vertex Splitting (TVSP)` | إيجاد أقل عدد رؤوس تقسّم الشجرة كي لا يتجاوز أي مسار تأخيراً محدداً `δ` | تصميم شبكات الاتصال |
| `delay d(T)` | أكبر تأخير مسار في الشجرة | `d(T) = max{d(p)}` |
| `Dijkstra's Algorithm` | إيجاد أقصر المسارات من مصدر واحد لكل الرؤوس بأوزان موجبة | `distance[], found[]` |
| `Relaxation` | تحديث تقدير مسافة رأس عند إيجاد مسار أقصر عبره | `distance[w]=distance[u]+cost[u][w]` |
| `Optimal Merge Pattern` | ترتيب دمج ملفات متعددة لتقليل إجمالي عدد التحركات | دمج أصغر ملفين أولاً |
| `Weighted External Path Length` | مجموع أطوال × أعماق الملفات في شجرة الدمج | `Σdᵢqᵢ` |
| `Spanning Tree` | بيان جزئي بلا حلقات يغطي كل رؤوس البيان الأصلي | `n-1` ضلع بالضبط |
| `Minimum-Cost Spanning Tree` | شجرة امتداد بأقل مجموع أوزان أضلاع | حلّها بـ `Kruskal`/`Prim` |

### المكونات الرئيسية (مرجع سريع)
| الأداة | الوظيفة | ملاحظة |
| --- | --- | --- |
| `Select(a)` | اختيار عنصر مرشح واحد من المدخلات | يحدد "الطمع" الفعلي للمسألة |
| `Feasible(solution, x)` | فحص جدوى إضافة عنصر للحل الجزئي | يمنع كسر القيود |
| `Union(solution, x)` | دمج عنصر مقبول في الحل الجاري بناؤه | لا رجوع بعد الدمج |
| `choose(distance, n, found)` | إيجاد أصغر مسافة بين رؤوس غير مثبتة | تكلفة `O(n)` |
| `Least(list)` | استخراج أصغر شجرة/عنصر من قائمة | `O(1)` مصفوفة أو `O(log n)` كومة |
| `Insert(list, t)` | إدراج عنصر جديد في القائمة | `O(n)` مصفوفة أو `O(log n)` كومة |

### جداول مقارنات سريعة
| المقارنة | الأول | الثاني | الفرق |
| --- | --- | --- | --- |
| `0/1 Knapsack` مقابل `Fractional Knapsack` | لا يقبل الكسر (`xᵢ ∈ {0,1}`) | يقبل الكسر (`xᵢ ∈ [0,1]`) | `greedy` أمثل فقط في النسخة الكسرية |
| تمثيل `list` بمصفوفة مقابل `heap` في `Tree()` | `O(n²)` إجمالاً | `O(n log₂n)` إجمالاً | فرق التعقيد الكلي للدمج المثلى |
| `Dijkstra` مقابل `TVS` | `O(n²)` | `Θ(n)` | `Dijkstra` يفحص كل الأزواج تقريباً، `TVS` يزور كل رأس مرة واحدة فقط |
| `Kruskal` مقابل `greedyKnapsack` (فكرة الترتيب) | يرتّب الأضلاع تصاعدياً | يرتّب الأصناف تنازلياً حسب `p/w` | كلاهما greedy لكن بمعيار ترتيب مختلف حسب المسألة |

### قاموس المصطلحات
| الفئة | المصطلحات |
| --- | --- |
| بنى بيانات | `tree_node`, `list`, `heap`, `adjacency matrix (cost[][])` |
| خوارزميات | `greedy`, `Dijkstra`, `Kruskal`, `Prim`, `TVS`, `Tree()` |
| قياسات | `distance`, `delay`, `weight`, `cost`, `p/w ratio` |
| رموز رياضية | `Σ`, `max`, `O(n)`, `O(n²)`, `O(n log₂n)`, `Θ(n)` |

### أبرز النقاط الذهبية
1. `Greedy` يعطي قراراً محلياً أفضل فقط — لا يضمن الأمثل العام دائماً إلا في مسائل محددة البنية (`Fractional Knapsack`, `Optimal Merge`, `MST`).
2. ترتيب المدخلات (`Select`) هو "قلب" أي خوارزمية `greedy` — تغييره قد يُحوّل الحل من أمثل إلى غير أمثل.
3. تعقيد أغلب خوارزميات `greedy` المذكورة يتحدد بتكلفة الترتيب المبدئي (`O(n log₂n)`) وليس بالحلقة الأساسية نفسها.
4. `Dijkstra` يعتمد على فرضية الأوزان الموجبة — وهذا شرط أساسي لصحته.
5. في `TVSP`، إعادة تصفير التأخير `d[T]=0` بعد القص يُحاكي "إعادة ضبط" مادية للتأخير المتراكم.

### الأخطاء الشائعة عند الطلاب ⚠️
| الخطأ | التصحيح |
| --- | --- |
| افتراض أن `greedy` يحل `0/1 Knapsack` بشكل أمثل | `greedy` أمثل فقط في `Fractional Knapsack`؛ `0/1` يحتاج `Dynamic Programming` |
| ترتيب أصناف Knapsack حسب الربح `pᵢ` فقط | الترتيب الصحيح حسب النسبة `pᵢ/wᵢ` |
| نسيان فحص الحلقة (`cycle`) عند بناء شجرة الامتداد | يجب التحقق من عدم تكوين حلقة قبل قبول أي ضلع |
| الاعتقاد أن تعقيد `Dijkstra` يعتمد على عدد الأضلاع | التعقيد الأساسي هنا `O(n²)` معتمد على عدد الرؤوس بهذا التمثيل |
| دمج أكبر ملفين أولاً في `Optimal Merge` | القاعدة الصحيحة: دمج أصغر ملفين أولاً في كل مرحلة |

### خطوات وإجراءات المحاضرة
> **كل عملية أو إجراء ورد في المحاضرة — مكتوب كـ `algorithm` block مستقل بالترتيب.**

#### ⚙️ الخطوات / الخوارزمية: General Greedy

> ما هدفه؟ بناء حل مرحلي عام لأي مسألة `greedy`.

```algorithm
1 | تهيئة solution | init | solution = Empty
2 | اختيار مرشح | Select(a) | اختيار x الأفضل محلياً
3 | فحص الجدوى | Feasible | تحقق من صلاحية إضافة x
4 | دمج | Union | ضم x المقبول إلى solution
5 | إرجاع | return | إعادة solution النهائي
```

#### نقاط التنفيذ:
- لا تراجع أبداً بعد قبول عنصر.

#### ⚙️ الخطوات / الخوارزمية: Greedy Knapsack

> ما هدفه؟ حل Fractional Knapsack بأخذ الأصناف مرتبة حسب p/w.

```algorithm
1 | ترتيب | sort | ترتيب تنازلي حسب p[i]/w[i]
2 | تهيئة U | init | U = m (السعة الكاملة)
3 | حلقة | loop | أخذ كل صنف كاملاً طالما w[i] <= U
4 | كسر أخير | fraction | x[i] = U/w[i] عند أول صنف لا يتسع
```

#### نقاط التنفيذ:
- الكسر يُؤخذ من صنف واحد فقط.

#### ⚙️ الخطوات / الخوارزمية: TVS

> ما هدفه؟ إيجاد أقل عدد رؤوس قصّ للحفاظ على تأخير أقل من δ.

```algorithm
1 | مرور post-order | recurse | معالجة كل الأبناء أولاً
2 | حساب d[T] | update | d[T]=max(d[T], d[child]+w(T,child))
3 | فحص الحد | check | d[T]+w(parent,T) > delta؟
4 | قص | split | إخراج T وتصفير d[T]
```

#### نقاط التنفيذ:
- الجذر لا يُفحص للقص.

#### ⚙️ الخطوات / الخوارزمية: Dijkstra Shortest Path

> ما هدفه؟ إيجاد أقصر مسار من رأس بداية لكل الرؤوس.

```algorithm
1 | تهيئة | init | distance[i]=cost[v][i]، found[i]=FALSE
2 | تثبيت البداية | base | distance[v]=0, found[v]=TRUE
3 | اختيار الأقرب | choose | u = أصغر distance بين غير المثبتين
4 | تحديث الجيران | relax | تحديث distance[w] إن وُجد مسار أقصر عبر u
```

#### نقاط التنفيذ:
- يتطلب أوزاناً موجبة فقط.

#### ⚙️ الخطوات / الخوارزمية: Tree() لبناء شجرة الدمج المثلى

> ما هدفه؟ دمج n ملف بأقل عدد تحركات إجمالي.

```algorithm
1 | تكرار n-1 مرة | loop | كرر حتى تبقى شجرة واحدة
2 | استخراج أصغر شجرتين | Least | مرتين متتاليتين
3 | دمج | merge | إنشاء أب جديد بوزن = مجموع وزني الابنين
4 | إعادة إدراج | Insert | إدراج الشجرة الجديدة في القائمة
```

#### نقاط التنفيذ:
- استخدام heap يُحسّن التعقيد إلى O(n log n).

#### ⚙️ الخطوات / الخوارزمية: مبدأ Kruskal التمهيدي

> ما هدفه؟ بناء شجرة امتداد بأقل كلفة.

```algorithm
1 | ترتيب الأضلاع | sort | تصاعدياً حسب الوزن
2 | اختيار | select | أرخص ضلع غير مُختار
3 | فحص الحلقة | check | تجنب تكوين حلقة
4 | توقف | stop | عند اختيار n-1 ضلعاً
```

#### نقاط التنفيذ:
- التفاصيل الكاملة (كود وتحليل) لم تُشرح في هذا الجزء من المحاضرة.

### أنماط الأكواد والبنى المتكررة
> كل بنية كود ظهرت أكثر من مرة أو تمثّل أسلوباً يجب أن يُحفظ.

| النمط | البنية الأساسية | متى تستخدمه |
| --- | --- | --- |
| `Greedy Loop` | `for(i=1;i<=n;i++){x=Select(a); if(Feasible) Union;}` | أي مسألة تحتاج بناء حل تدريجي بقرار محلي |
| `Relaxation Loop` | `if(distance[u]+cost[u][w] < distance[w]) distance[w]=...` | تحديث تقديرات المسافة في خوارزميات المسارات |
| `Post-order Recursion` | `recurse(child); update(parent) from child;` | حساب قيم تعتمد على الأبناء أولاً (TVS، الأشجار عموماً) |
| `Repeated Extract-Min` | `while(more than one item) { a=Least(list); b=Least(list); merge; Insert; }` | بناء أشجار Huffman/Merge المثلى |

### أنماط التعامل والسلوك
> كيف يتصرف النظام / الخوارزمية في سيناريوهات مختلفة.

| السيناريو | التعامل الصحيح | لماذا؟ |
| --- | --- | --- |
| صنف Knapsack وزنه يفوق السعة المتبقية تماماً | يُؤخذ منه كسر فقط `x[i]=U/w[i]` ثم تتوقف الحلقة | لضمان ملء الحقيبة بالكامل دون تجاوز m |
| رأس في TVS تجاوز تأخيره δ بعد ربطه بالأب | يُخرَج كنقطة قص ويُصفَّر تأخيره | لإبقاء المسارات اللاحقة ضمن الحد المسموح |
| رأس في Dijkstra تم تثبيته (found=TRUE) | لا تُعاد معالجته أبداً في الحلقات اللاحقة | لأن مسافته النهائية لن تتغير (أوزان موجبة) |
| دمج ملفات بأحجام متعددة | ادمج أصغر ملفين متاحين في كل مرحلة | لتقليل إجمالي عدد التحركات (weighted external path) |

### الأفكار الرئيسية الشاملة
> أي فكرة محورية لم تُغطَّ أعلاه.

- **العلاقة بين كل هذه الخوارزميات**: جميعها (Knapsack، TVS، Dijkstra، Optimal Merge، MST) هي **تطبيقات مختلفة لنفس الإطار العام** (`greedy`)، تختلف فقط في: (1) دالة `Select` المستخدمة، و(2) شرط `Feasible` الخاص بكل مسألة.
- ذُكر أيضاً في نهاية الفصل (كملحق تنفيذي منفصل غير مرتبط مباشرة بالتسلسل النظري أعلاه) خوارزمية بسيطة لتخصيص الذاكرة (`Memory Block Allocation`) تجد لكل عملية أصغر كتلة ذاكرة حرة تكفيها — وهي تطبيق آخر لنفس فكرة `greedy` (اختيار أصغر خيار مناسب بدل البحث الشامل).

```algorithm
1 | إدخال البيانات | input | إدخال كتل الذاكرة والعمليات بأحجامها
2 | تهيئة | init | تعليم كل كتل الذاكرة كحرة
3 | اختيار كتلة | select | لكل عملية، إيجاد أصغر كتلة حرة تكفيها
4 | تخصيص | assign | إذا وُجدت كتلة مناسبة، خصّصها للعملية الحالية
5 | التخطي | skip | إذا لم تُوجد كتلة مناسبة، انتقل للعملية التالية
```

## الجزء الرابع: أسئلة تصحيح الكود

## أسئلة تصحيح الكود
> غطِّ أنواع الأخطاء: ترجمة، منطقية، فحص إرجاع، dead code، سوء فهم.

### سؤال تصحيح 1 — Logic

**الكود (يحتوي خطأ):**
```cpp
void greedyKnapsack(float m, int n) {
    for (int i = 1; i <= n; i++) x[i] = 0.0;
    float U = m;
    int i;
    for (i = 1; i <= n; i++) {
        if (w[i] > U) break;
        x[i] = 1.0;
        // خطأ: نسي تحديث U
    }
    if (i <= n) x[i] = U / w[i];
}
```
**اكتشف الخطأ:** لم يتم تحديث `U -= w[i]` بعد أخذ كل صنف كاملاً، فتبقى `U` تحمل السعة الكاملة طوال التنفيذ.

**التصحيح:**
```cpp
void greedyKnapsack(float m, int n) {
    for (int i = 1; i <= n; i++) x[i] = 0.0;
    float U = m;
    int i;
    for (i = 1; i <= n; i++) {
        if (w[i] > U) break;
        x[i] = 1.0;
        U -= w[i]; // تحديث صحيح للسعة المتبقية
    }
    if (i <= n) x[i] = U / w[i];
}
```
**شرح الحل:**
1. `U` تمثل السعة **المتبقية**، ويجب إنقاصها بعد كل أخذ كامل.
2. بدون التحديث، ستأخذ الخوارزمية كل الأصناف كاملة دون فحص السعة الحقيقية، مخالفة القيد `Σwᵢxᵢ ≤ m`.
3. النتيجة الخاطئة تظهر عادة كوزن كلي أكبر من `m` أو كسر خاطئ في النهاية.

### سؤال تصحيح 2 — Misconception

**الكود (يحتوي خطأ):**
```cpp
// ترتيب الأصناف تصاعدياً حسب الوزن قبل تشغيل greedyKnapsack
sort_ascending_by_weight(p, w, n);
greedyKnapsack(m, n);
```
**اكتشف الخطأ:** سوء فهم لمعيار الترتيب الصحيح — يجب الترتيب حسب النسبة `p[i]/w[i]` تنازلياً، وليس حسب الوزن وحده تصاعدياً.

**التصحيح:**
```cpp
// ترتيب الأصناف تنازلياً حسب النسبة p[i]/w[i]
sort_descending_by_ratio(p, w, n);
greedyKnapsack(m, n);
```
**شرح الحل:**
1. الترتيب حسب الوزن فقط قد يختار أصنافاً خفيفة لكن ذات ربح ضعيف قبل أصناف أثقل وأكثر ربحاً نسبياً.
2. المعيار الأمثل مُثبت رياضياً بالمثال العددي: النسبة `p/w` الأعلى تعطي أعلى ربح ممكن.
3. الترتيب الخاطئ لا يُسبب خطأً في التنفيذ (الكود يعمل) لكنه يعطي **حلاً غير أمثل**.

### سؤال تصحيح 3 — Return Check

**الكود (يحتوي خطأ):**
```cpp
int choose(int distance[], int n, short int found[]) {
    int min = INFTY;
    int minpos = 0;
    for (int i = 1; i <= n; i++)
        if (distance[i] < min) {   // خطأ: نسي فحص found[i]
            min = distance[i];
            minpos = i;
        }
    return minpos;
}
```
**اكتشف الخطأ:** الشرط لا يتحقق من `found[i]==FALSE`، فقد يُعاد رأس تم تثبيته مسبقاً، مما يُفسد الخوارزمية (حلقة لا نهائية أو نتائج خاطئة).

**التصحيح:**
```cpp
int choose(int distance[], int n, short int found[]) {
    int min = INFTY;
    int minpos = 0;
    for (int i = 1; i <= n; i++)
        if (distance[i] < min && found[i] == FALSE) { // فحص إضافي صحيح
            min = distance[i];
            minpos = i;
        }
    return minpos;
}
```
**شرح الحل:**
1. `choose()` يجب أن تبحث فقط بين الرؤوس **غير المثبتة بعد**.
2. بدون هذا الفحص، قد يُعاد نفس الرأس مراراً فيتعطل تقدّم الحلقة الرئيسية في `shortest_path()`.
3. هذا مثال كلاسيكي على أهمية فحص القيم المرجعة/الشروط المنطقية المصاحبة (`found[]`) وليس فقط القيمة العددية (`distance[]`).

### سؤال تصحيح 4 — Dead Code

**الكود (يحتوي خطأ):**
```cpp
void TVS(tree_pointer T, Type delta) {
    if (T != NULL) {
        d[T] = 0;
        for (/* each child v of T */) {
            TVS(v, delta);
            d[T] = max(d[T], d[v] + w(T, v));
        }
        if ((T is not the root) && (d[T] + w(parent(T), T) > delta)) {
            cout << T << endl;
            d[T] = 0;
        }
        return; // خطأ: سطر ميت لا فائدة له هنا لكنه غير ضار مباشرة، لكن الخطأ الحقيقي التالي:
        d[T] = -1; // كود ميت لا يُنفَّذ أبداً بعد return
    }
}
```
**اكتشف الخطأ:** السطر `d[T] = -1;` بعد `return` هو **كود ميت (dead code)** لن يُنفَّذ أبداً، وهو أثر من تعديل سابق لم يُحذف.

**التصحيح:**
```cpp
void TVS(tree_pointer T, Type delta) {
    if (T != NULL) {
        d[T] = 0;
        for (/* each child v of T */) {
            TVS(v, delta);
            d[T] = max(d[T], d[v] + w(T, v));
        }
        if ((T is not the root) && (d[T] + w(parent(T), T) > delta)) {
            cout << T << endl;
            d[T] = 0;
        }
    }
}
```
**شرح الحل:**
1. أي سطر بعد `return` غير المشروط في نفس الكتلة لن يُنفَّذ أبداً — يجب حذفه فوراً.
2. الكود الميت لا يُسبب خطأً وظيفياً هنا لكنه يُربك القارئ ويوحي بمنطق غير موجود فعلياً.
3. القاعدة العامة: أي كود يقع بعد `return`/`break`/`continue` غير المشروط في نفس نطاق التنفيذ يجب مراجعته وحذفه إن كان لا يصل إليه التنفيذ أبداً.

### سؤال تصحيح 5 — Off-by-one

**الكود (يحتوي خطأ):**
```cpp
Type *Tree(int n) {
    for (int i = 1; i <= n; i++) {   // خطأ: يجب أن تكون i < n وليس i <= n
        Type *pt = new Type;
        pt->l_child = Least(list);
        pt->r_child = Least(list);
        pt->weight = (pt->l_child)->weight + (pt->r_child)->weight;
        Insert(list, *pt);
    }
    return (Least(list));
}
```
**اكتشف الخطأ:** الحلقة تُنفَّذ `n` مرة بدلاً من `n-1` مرة، مما يسبب استدعاء إضافي لـ `Least(list)` على قائمة فارغة أو غير متوقعة في التكرار الأخير (خطأ `off-by-one`).

**التصحيح:**
```cpp
Type *Tree(int n) {
    for (int i = 1; i < n; i++) {    // صحيح: n-1 تكرار فقط
        Type *pt = new Type;
        pt->l_child = Least(list);
        pt->r_child = Least(list);
        pt->weight = (pt->l_child)->weight + (pt->r_child)->weight;
        Insert(list, *pt);
    }
    return (Least(list));
}
```
**شرح الحل:**
1. لدمج `n` شجرة في شجرة واحدة نحتاج بالضبط `n-1` عملية دمج (كل عملية دمج تُنقص عدد الأشجار في `list` بمقدار واحد).
2. تنفيذ الحلقة `n` مرة يعني محاولة دمج إضافي بعد أن تبقى شجرة واحدة فقط في `list`، وهذا يُسبب سلوكاً غير معرّف عند استدعاء `Least` على عنصر واحد متبقٍ أو قائمة فارغة.
3. هذا خطأ `off-by-one` شائع جداً عند تحويل "عدد العناصر" إلى "عدد الدمجات المطلوبة".

## بطاقات سؤال وجواب (Q&A Cards)

**Q1:** ما هو `Greedy Method`؟
A: استراتيجية بناء حل تدريجياً باختيار الأفضل محلياً في كل خطوة دون رجوع.

**Q2:** ما الفرق بين `Feasible` و`Union` في القالب العام لـ greedy؟
A: `Feasible` تتحقق من صحة إضافة عنصر، و`Union` تدمجه فعلياً في الحل بعد قبوله.

**Q3:** ما معيار الترتيب الصحيح في Fractional Knapsack؟
A: الترتيب التنازلي حسب النسبة `p/w` (الربح لكل وحدة وزن).

**Q4:** هل `greedy` يحل `0/1 Knapsack` بشكل أمثل؟
A: لا، فقط النسخة الكسرية (Fractional Knapsack) تُحل بشكل أمثل بـ greedy.

**Q5:** ما هو `delay d(T)` لشجرة؟
A: أكبر تأخير `d(p)` بين كل المسارات الممكنة داخل الشجرة `T`.

**Q6:** ما هدف خوارزمية `TVS`؟
A: تحديد أقل عدد رؤوس قصّ في شجرة موزونة بحيث لا يتجاوز أي مسار حداً `δ`.

**Q7:** ما الشرط الأساسي لصحة خوارزمية Dijkstra؟
A: أن تكون كل أوزان الأضلاع موجبة (`w(e) > 0`).

**Q8:** ما تعقيد خوارزمية Dijkstra بالتمثيل المصفوفي المذكور؟
A: `O(n²)`.

**Q9:** ما استراتيجية greedy الصحيحة في Optimal Merge Patterns؟
A: دمج أصغر ملفين (أو شجرتين) متاحين في كل مرحلة.

**Q10:** ما هو `weighted external path length`؟
A: مجموع (عمق كل ملف في شجرة الدمج × حجمه)، ويمثل إجمالي عدد تحركات السجلات.

**Q11:** ما الفرق بين تمثيل list بمصفوفة و heap في دالة Tree()؟
A: المصفوفة تعطي تعقيداً كلياً O(n²)، بينما الـ heap يخفضه إلى O(n log n).

**Q12:** كم عدد أضلاع شجرة الامتداد لبيان بـ n رأس؟
A: n-1 ضلعاً بالضبط.

**Q13:** ما القيدان الآخران (غير عدد الأضلاع) لشجرة الامتداد؟
A: الأضلاع من البيان الأصلي فقط، ولا يوجد ضلع يُشكّل حلقة.

**Q14:** ما اسم الخوارزميتين المذكورتين لحل Minimum-Cost Spanning Tree؟
A: خوارزمية Kruskal وخوارزمية Prim.

### سؤال 1: عرّف The Greedy Method واذكر مكوناته الأساسية.
**نموذج الإجابة:**
1. التعريف: استراتيجية حل مسائل التحسين ببناء الحل تدريجياً عبر قرارات محلية أفضل في كل خطوة، دون تراجع.
2. المكونات/الشروط: `Select` (اختيار مرشح)، `Feasible` (فحص الجدوى)، `Union` (الدمج بالحل الجزئي)، `Objective Function` (دالة الهدف المراد تعظيمها/تصغيرها).
3. مثال: `Fractional Knapsack` باختيار أعلى نسبة `p/w`.
4. متى نستخدم: عندما يكون للمسألة بنية تضمن أن القرارات المحلية المثلى تؤدي لحل عام أمثل أو مقبول (خصائص matroid أو ما شابه).

### سؤال 2: قارن بين Fractional Knapsack و0/1 Knapsack من حيث قابلية الحل بـ greedy.
**نموذج الإجابة:**
1. التعريف: الأولى تسمح بأخذ كسر من الصنف (`xᵢ∈[0,1]`)، والثانية تُلزم بأخذ الصنف كاملاً أو تركه (`xᵢ∈{0,1}`).
2. المكونات/الشروط: كلتاهما تشترك في نفس دالة الهدف والقيد على الوزن الكلي.
3. مثال: مثال المحاضرة (`m=20`) يوضح كيف يعطي `greedy` حلاً أمثل (`31.5`) في النسخة الكسرية.
4. متى نستخدم: `greedy` للنسخة الكسرية فقط؛ النسخة الصحيحة (0/1) تحتاج `Dynamic Programming` لضمان الأمثلية.

### سؤال 3: اشرح مفهوم `delay` في مسألة Tree Vertex Splitting وكيف يُحسب.
**نموذج الإجابة:**
1. التعريف: `delay` هو التأخير المتراكم لمسار من رأس داخل الشجرة إلى الجذر (أو العكس حسب الاتجاه)، ويُرمز له بـ `d(u)`.
2. المكونات/الشروط: يُحسب بمعادلة `d(u) = max{d(w)+w(w,u) : w من أبناء u}`، والورقة `d=0`.
3. مثال: مثال المحاضرة (`δ=5`) حيث `d(4)=0` ثم فحص `d(2)+w(2,4)>δ`.
4. متى نستخدم: عند تصميم شبكات ذات حد أقصى مسموح للتأخير ونريد أقل عدد نقاط "إعادة ضبط".

### سؤال 4: اشرح خطوات خوارزمية Dijkstra ولماذا تُصنَّف كـ greedy.
**نموذج الإجابة:**
1. التعريف: خوارزمية لإيجاد أقصر مسار من رأس بداية واحد لكل بقية رؤوس بيان موزون بأوزان موجبة.
2. المكونات/الشروط: `distance[]`, `found[]`, `choose()` لاختيار أصغر مسافة غير مثبتة، وخطوة `relaxation`.
3. مثال: مثال الشكل (٣-٤) حيث أقصر مسار من 1 إلى 2 كان عبر 3 عدد مسافات أقل من الضلع المباشر.
4. متى نستخدم: عند الحاجة لأقصر مسارات من مصدر واحد لكل الوجهات في بيان بأوزان موجبة فقط. تُصنَّف كـ greedy لأنها تختار في كل خطوة أقرب رأس غير مثبت وتُثبّته نهائياً دون رجوع.

### سؤال 5: ما هي استراتيجية Optimal Merge Pattern ولماذا تعمل؟
**نموذج الإجابة:**
1. التعريف: طريقة لترتيب عمليات دمج عدة ملفات مرتّبة لتقليل إجمالي عدد التحركات.
2. المكونات/الشروط: تعتمد على دمج أصغر ملفين (أو شجرتين) في كل مرحلة، وتُستخدم دالة `Tree()` مع `Least`/`Insert`.
3. مثال: مثال المحاضرة لخمس ملفات (5,30,20,10,30) بإجمالي 205 تحركاً.
4. متى نستخدم: عند دمج أكثر من ملفين مرتّبين ونريد أقل تكلفة نسخ/مقارنة إجمالية.

### سؤال 6: ما تعريف Weighted External Path Length وعلاقتها بتكلفة الدمج؟
**نموذج الإجابة:**
1. التعريف: مجموع حاصل ضرب عمق كل عقدة طرفية في شجرة الدمج بحجمها (`Σdᵢqᵢ`).
2. المكونات/الشروط: تعتمد على شكل شجرة الدمج الناتجة عن ترتيب عمليات الدمج.
3. مثال: في شجرة المحاضرة، إجمالي التحركات = 205، وهو نفس قيمة الـ weighted external path length لتلك الشجرة.
4. متى نستخدم: كمقياس مباشر لمقارنة كفاءة استراتيجيات دمج مختلفة قبل تنفيذها فعلياً.

### سؤال 7: عرّف شجرة الامتداد (Spanning Tree) واذكر قيودها.
**نموذج الإجابة:**
1. التعريف: بيان جزئي `t=<V,É>` من بيان أصلي `G=<V,E>` يُشكّل شجرة (بلا حلقات) تغطي كل رؤوس `G`.
2. المكونات/الشروط: الأضلاع من `G` فقط، عددها `n-1` بالضبط، ولا حلقات.
3. مثال: الشكل (٣-٧) في المحاضرة يوضح أربع أشجار امتداد مختلفة لبيان كامل واحد.
4. متى نستخدم: عند الحاجة لربط كل عناصر شبكة بأقل عدد اتصالات ممكن دون تكرار.

### سؤال 8: لماذا تُصنَّف خوارزميتا Kruskal وPrim ضمن أسلوب greedy لحل مسألة MST؟
**نموذج الإجابة:**
1. التعريف: كلتاهما تختاران في كل خطوة الضلع الأرخص المتاح (وفق معيار كل خوارزمية) دون التراجع لاحقاً.
2. المكونات/الشروط: تستخدمان معيار "الأقل كلفة" (`least cost`)، وتتحققان من قيود شجرة الامتداد (لا حلقات، n-1 ضلع).
3. مثال: مبدأ Kruskal التمهيدي المذكور: ترتيب كل الأضلاع تصاعدياً ثم اختيار الأرخص غير المُكوّن لحلقة.
4. متى نستخدم: عند بناء شبكة اتصال بأقل تكلفة كلية بين مجموعة عقد.

## ورقة المراجعة السريعة (Cheat Sheet)

### 🔑 خريطة العلاقات بين المحاضرات
| المحاضرة | ترتبط مع | كيف؟ |
| --- | --- | --- |
| General Greedy | كل الأمثلة اللاحقة | جميعها تطبيقات لنفس القالب (`Select/Feasible/Union`) |
| Knapsack | Optimal Merge / MST | كلها تستخدم معيار ترتيب مختلف يحقق أمثلية greedy لكل مسألة |
| TVS | Dijkstra | كلاهما يعتمد على تحديث/حساب قيم تراكمية على بيان/شجرة (delay مقابل distance) |
| Optimal Merge | MST (لاحقاً) | كلاهما يستخدم بنية شجرة و/أو heap لتقليل التكلفة الكلية |

### 🔑 أهم النقاط الذهبية
| الموضوع | النقاط |
| --- | --- |
| Greedy عام | قرار محلي أفضل + لا رجوع؛ لا يضمن الأمثل العام دائماً |
| Knapsack | ترتيب تنازلي حسب `p/w`؛ أمثل فقط في النسخة الكسرية |
| TVS | `d(u)=max(d(w)+w(w,u))`؛ قص عند تجاوز `δ`؛ تعقيد `Θ(n)` |
| Dijkstra | أوزان موجبة فقط؛ `relaxation`؛ تعقيد `O(n²)` |
| Optimal Merge | دمج أصغر شجرتين أولاً؛ heap يُحسّن التعقيد إلى `O(n log n)` |
| MST | `n-1` ضلع، لا حلقات، أضلاع من G فقط؛ Kruskal/Prim كأدوات حل |

### 🔑 مرجع سريع
| الرمز/المصطلح | المعنى | يُستخدم في |
| --- | --- | --- |
| `p/w` | نسبة الربح للوزن | Knapsack |
| `d(T)`, `δ` | التأخير الأقصى، الحد المسموح | TVSP |
| `distance[]`, `found[]` | مسافة تقديرية، هل ثُبّتت | Dijkstra |
| `Least(list)`, `Insert(list,t)` | استخراج/إدراج أصغر عنصر | Optimal Merge (Tree()) |
| `n-1` | عدد أضلاع شجرة الامتداد | MST |

### 🔑 قواعد ذهبية لا تُنسى
| # | القاعدة |
| --- | --- |
| 1 | Greedy لا يعيد النظر أبداً في قرار سابق. |
| 2 | معيار الترتيب هو ما يُحدد نجاح greedy في كل مسألة. |
| 3 | 0/1 Knapsack ≠ Fractional Knapsack من حيث أمثلية greedy. |
| 4 | Dijkstra يتطلب أوزاناً موجبة حصراً. |
| 5 | دمج الأصغر أولاً دائماً في Optimal Merge Pattern. |
| 6 | شجرة الامتداد دائماً بها n-1 ضلع بالضبط، بلا حلقات. |
